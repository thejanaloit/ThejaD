Memory locking (how it works)

  • Pod LWW lock — thejad_pod_memory_sync merges LAN peers; per key, the entry with the newest at timestamp wins (last-write-wins).
  • Coordination claim lock — coordination_claim / coordination_release locks file lanes so Cursor, Copilot, and Antigravity do not collide.
  • Orchestra snapshot — runOrchestra calls podMemorySync() first so all models read the same locked graphical memory before voting.
  • Tier lock — mamaThejana → thejad_unlock holds 100% tier until nothejad unlock.

Graphify index

graphify_hint → graphify . → graphify-out/. Query architecture (apps, Nest services, migrations) as a graph — fewer tokens than reading every file.

Mesh-linked memory + locking

Pod shared-memory uses LWW lock (newest at wins on sync). Coordination claim locks edit lanes. Orchestra reads one locked graph snapshot — not diverging chat paste.

Setup → orchestrate flow

Setup builds graphical memory (Graphify index + mesh sync), then every prompt hits thejad_orchestrate — models warm, the graph is read, role/skills/agent picked, token-efficient plan, delivery workflow.
